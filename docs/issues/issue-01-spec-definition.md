# 仕様策定

## 概要・背景・目的

otegami の設計を確定し、実装の土台となる文書群を整える。テキストを一度だけ託して別の場所で受け取る最小サービスとして、目的・原則・スコープ・セキュリティ・削除モデルを文書に固定し、以降の issue の唯一の参照源(SSoT)とする。

## 受け入れ条件

- [ ] design-spec.md がある(目的・二原則・スコープ・利用フロー・アーキテクチャ・データモデル・削除モデル・セキュリティ・実行時設定・確定判断)
- [ ] README.md がある(概要・特徴・使い方・技術スタック・ディレクトリ構成・ドキュメント構成)
- [ ] ADR-0001〜0004 がある(セキュリティモデル・cleaner 分離・TTL+burn・Cloudflare Access)
- [ ] 未決事項が解消され、design-spec に反映されている

## 技術的な検討事項

- 文書は docs/ に三層構成(README / design-spec / adr)で置く
- 実行時設定(TTL・slug 長など)は `.bootstrap.toml` ではなくアプリの設定として扱う(design-spec 10章)

## 関連ADR・依存issue

- ADR: 0001, 0002, 0003, 0004
- 依存: なし(最初の issue)

## 想定工数・優先度

- 優先度: 最高(他のすべての前提)
- 状態: 成果物は作成済み。内容を確認のうえクローズ可
