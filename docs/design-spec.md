# otegami — 設計仕様書 (design-spec)

## このドキュメントについて

otegami(お手紙)は、ログインできない環境からでも、テキストを一通の手紙のように託して別の場所で受け取るための、最小構成のテキスト共有サービスである。本書は実装に先立つ設計判断を一枚にまとめ、以降の README / ADR / issues の出発点とする。

## 1. 背景と目的

自分のアカウントにログインできない貸与PCなどから、作業途中のコードやAI出力などのテキストを、別の環境へ持ち出したい場面がある。チャットやメールに貼るには長く、専用ツールも入れられない。

そこで「テキストを貼る → 短いURLとパスワードを得る → 別の場所でそれを開いて受け取る」という、託して受け取るだけの導線を提供する。原則として運用者一人の利用を想定するが、URLとパスワードを知る者は誰でも閲覧できる。

## 2. 設計の二原則

- 主体性をユーザーに返す。残す/消す/守るの判断は作成者が握る。サービスが勝手に永続化したり、勝手に晒したりしない。
- 認識は丁寧に正確に渡す。貼ったテキストは改変も解釈もせず、そのままの文字として正確に届ける。サービスはあくまで運び手に徹する。

## 3. スコープ

### やること

- テキストの登録(タイトル任意・本文必須)
- ランダムで短い、手打ち可能なURLの発行
- 閲覧時のパスワード確認
- 「閲覧後に削除」(burn after reading)の指定
- 既定TTLによる自動失効
- 10分間隔のバッチによる失効分・閲覧済み分の物理削除

### やらないこと(非目標)

- 作成後の編集。otegami は一度きりの手紙であり、編集は持たない
- リッチテキスト/Markdown描画。本文は素のテキストとして扱う(URL自動リンクは将来の任意拡張)
- ユーザーアカウント機能。作成導線の本人確認は Cloudflare Access に委ね、アプリ内に認証コードを持たない
- 閲覧数カウントや解析。運び手に不要な記憶は持たせない
- ファイル添付・複数本文

UNIX哲学に倣い、「テキストを安全に一度運ぶ」一点に責務を絞る。

## 4. 利用フロー

作成(運用者のみ。Cloudflare Access で保護):

1. 本文(任意でタイトル)とパスワードを入力し、必要なら「閲覧後に削除」を選ぶ
2. サーバがランダムなslugを発行し、本文を保存、パスワードはハッシュ化して保存
3. `https://<host>/n/<slug>` を提示する

閲覧(URLとパスワードを知る者なら誰でも):

1. `/n/<slug>` を開くとパスワード入力フォームが出る(slugの存在有無は明かさない)
2. パスワードをPOSTで送る。slugとパスワードの双方が一致した場合のみ本文を表示する
3. 不一致時は「見つからないか、パスワードが違う」という単一のエラーを返す
4. 表示に成功した時点で `viewed_at` を記録する

## 5. アーキテクチャ

クリーンアーキテクチャに沿い、内側(ドメイン)が外側(Web/DB)を知らない依存方向を保つ。

### レイヤー構成

- domain: エンティティと値オブジェクト、ドメインエラー。フレームワークを一切知らない
- application: ユースケース。ポート(trait)経由でのみ外部に触れる
- infrastructure: ポートの具体実装(Postgres・Argon2・nanoid・時計)
- presentation: Axum ハンドラと Askama テンプレート。掃除バッチは別バイナリ

### ユースケース

- CreateNote: 本文・パスワード・burnフラグを受け取り、slugを採番して保存する
- ViewNote: slugとパスワードを照合し、一致時に本文を返し `viewed_at` を立てる
- PurgeNotes: 失効分・閲覧済みburn分を物理削除する(バッチが駆動)

### ポート(trait)

- NoteRepository: 保存・slug採番付き登録・slug検索・viewed記録・条件削除
- PasswordHasher: ハッシュ化と照合(Argon2 実装)
- SlugGenerator: 衝突しにくい短いslug生成(nanoid 実装)
- Clock: 現在時刻(テスト可能性のため抽象化)

外側の実装を差し替えても、ドメインとユースケースは無傷であることを設計目標とする。

## 6. データモデル

単一テーブル `notes` で完結させる。

```sql
CREATE TABLE notes (
    id                BIGSERIAL    PRIMARY KEY,        -- 内部ID。外部には露出しない
    slug              TEXT         NOT NULL UNIQUE,    -- 公開ハンドル
    title             TEXT,                            -- 任意
    body              TEXT         NOT NULL,            -- 素のテキストとして保存
    password_hash     TEXT         NOT NULL,            -- Argon2
    burn_after_view   BOOLEAN      NOT NULL DEFAULT FALSE,
    viewed_at         TIMESTAMPTZ,                      -- 初回閲覧成功時刻
    expires_at        TIMESTAMPTZ  NOT NULL,            -- 既定TTLから算出
    created_at        TIMESTAMPTZ  NOT NULL DEFAULT now()
);

CREATE INDEX idx_notes_slug ON notes (slug);
CREATE INDEX idx_notes_purge ON notes (expires_at);
```

外部に出るのは `slug` だけで、連番 `id` は秘匿する(列挙耐性のため)。

## 7. 削除モデル

全てのノートは作成から1日(既定TTL)で自動失効する。これが共通の安全網であり、burn を指定しない限り、その1日のあいだは何度でも読み返せる。読み返す必要がなく早く消したい手紙にだけ「閲覧後に削除」を指定する、という二段構えにする。

物理削除は10分間隔のバッチ(独立バイナリ `otegami-cleaner`)が担う。削除条件は次のいずれか。

- `expires_at < now()`(既定TTL切れ)
- `burn_after_view = TRUE AND viewed_at IS NOT NULL`(閲覧済みのburn分)

burn指定でも削除は即時ではなく次バッチで起きるため、閲覧から最大10分の猶予が残る。この緩さは運用者一人を想定した割り切りであり、即時消去が要件化したら閲覧成功時の同期削除へ拡張する。

掃除をアプリ本体から切り出す理由は、単一責任と疎結合のため。Web プロセスの生死と掃除の駆動を独立させ、cron / systemd-timer / Coolify のスケジュールいずれからでも叩けるようにする。

## 8. セキュリティ設計

防御は多層で構成し、どれか一枚に依存しない。

- 第一の鍵はURL。推測困難なslugそのものが事実上のアクセス制御となる
- 第二の軽い鍵はパスワード。手打ちのしやすさを優先し、半角英数1〜4文字に制限する。短さは利便性のための割り切りであり、安全はslugの推測困難さとレート制限で担保する
- レート制限。同一IP・同一slugに対する閲覧試行を制限し、短いパスワードへの総当たりを実質的に止める(これが短いパスワードを許せる前提条件)
- パスワードはArgon2でハッシュ化して保存し、平文は持たない
- 列挙・存在オラクルの抑止。閲覧失敗は「見つからない」と「パスワード違い」を区別しない単一エラーにする
- XSSの遮断。本文は素のテキストとして保存し、表示は Askama の自動エスケープに委ねる。`|safe` は使わない。本文は実行されず文字として描かれる
- 作成導線の保護。`/n/...` の閲覧は公開のままとし、作成系エンドポイントのみ Cloudflare Access で本人に限定する。アプリ内に認証コードを持ち込まない
- 経路の暗号化は Cloudflare Tunnel に委ねる

## 9. 技術スタック

- 言語: Rust
- Web: Axum(tower / tower-http のミドルウェア合成を利用)
- テンプレート: Askama(コンパイル時生成・自動エスケープ)
- DB: PostgreSQL + SQLx
- ハッシュ: argon2
- slug生成: Crockford Base32 系の6文字(人の書き写しを前提とした文字集合で、大文字小文字を区別せず I/L/O/U を除く。短さと記憶のしやすさを優先)
- レート制限: tower-governor 等のミドルウェア
- ログ: tracing
- 作成導線の認証: Cloudflare Access
- 配信/暗号化: Cloudflare Tunnel
- デプロイ: Coolify(A1)
- 構成: Web 本体 `otegami` と掃除 `otegami-cleaner` の二バイナリ

## 10. アプリの実行時設定

以下はアプリ自身が起動時に読む設定であり、`.bootstrap.toml`(project-bootstrap への指示書)とは別物である。`.bootstrap.toml` はプロジェクト名と stack を宣言するだけで、これらの値は持たない。実体はアプリの設定ファイル(例: `config.toml`)や環境変数として与える。

- データベース接続情報
- 既定TTL(初期値1日)
- slug長(初期値6文字)
- パスワード最大長(初期値4文字)
- レート制限の閾値(試行回数・期間)
- バッチ間隔(初期値10分)

## 11. 確定した主な判断

- 「閲覧後に削除」は作成時の任意フラグとする。既定TTL(1日)と組み合わせ、burn 未指定なら1日は読み返せる
- 既定TTLは作成から1日とする
- 短いURLの発行を必須機能とし、URL自動リンクはv1に含めず将来拡張に回す
- slug長は6文字とする

## 付録: 用語

- slug: 公開URLの末尾に置く、ノートを一意に指す短い乱数文字列
- burn after reading: 閲覧されたら次バッチで削除される指定
- TTL: ノートが自動失効するまでの寿命
- ポート: ユースケースが外部実装に触れるための抽象(trait)境界
