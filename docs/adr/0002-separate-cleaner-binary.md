# ADR-0002: 掃除処理を独立したバイナリに分離する

- ステータス: 受理
- 日付: 2026-06-24

## 文脈

失効したノートと、閲覧済みの burn 分は、定期的に物理削除する必要がある。駆動方法には、Web プロセス内に常駐タスク(tokio interval 等)を持たせる方式と、独立したバイナリを外部スケジューラから叩く方式がある。

otegami の責務は「テキストを安全に一度運ぶ」一点にある。掃除はそれとは別の関心であり、Web の応答処理やプロセスの寿命に結びつける必然性はない。

## 決定

- 掃除は `otegami-cleaner` という独立バイナリとして実装し、Web 本体 `otegami` と分離する。
- 駆動は外部スケジューラ(cron / systemd-timer / Coolify のスケジュール)に委ね、既定間隔は10分とする。
- 両バイナリはドメイン層とアプリケーション層を共有し、削除条件は PurgeNotes ユースケースに集約して二重定義を避ける。

## 結果

- 利点: 単一責任と疎結合を保てる。Web の再起動やスケールと、掃除の駆動が互いに独立する。掃除だけを手動実行して検証できる。UNIX哲学に沿う。
- トレードオフ: デプロイ対象が二つになり、スケジューラ設定という運用要素が増える。
- 緩さの容認: バッチ間隔ぶん(最大10分)は削除が遅延する。運用者一人の用途では許容する。

## 考慮した代替案

- Web プロセス内の常駐タスク: デプロイは単純になるが、Web の生死に掃除が縛られ、関心が混ざる。初期の手軽さを優先して将来 interval 版へ寄せる余地は残すが、既定は分離とする。
