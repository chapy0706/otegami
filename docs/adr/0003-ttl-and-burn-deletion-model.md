# ADR-0003: TTL と burn を組み合わせた削除モデル

- ステータス: 受理
- 日付: 2026-06-24

## 文脈

当初は「閲覧後に削除(burn after reading)」のみを検討した。しかし burn 単独では、一度開いた手紙を後で見返したい場面で困る。

一方で、何も消えない設計は、不要になった秘密が残り続けてしまう。運用者一人の持ち出し用途では、たいてい当日中に受け取りが完了するという前提もある。

## 決定

- すべてのノートに、作成からの既定TTL(初期値1日)を必ず設定する。これを共通の安全網とする。
- 「閲覧後に削除」は作成時の任意フラグとする。指定しなければ、TTL満了まで(=1日)は何度でも読み返せる。
- 物理削除の条件は、TTL満了、または burn 指定かつ閲覧済み、のいずれか。削除は次の掃除バッチで起きる。

## 結果

- 利点: 「見返したい」と「残したくない」を両立できる。読み返しは TTL 内で自由、即時性が要る手紙にだけ burn を足す二段構えになる。放置されたノートも必ず1日で消える。
- トレードオフ: burn を指定しても削除は即時ではなく次バッチ(最大10分の猶予)で起きる。一人運用では許容する。
- 拡張余地: 即時消去や TTL の選択UIが要件化したら、本ADRを置換する新しいADRで対応する。

## 考慮した代替案

- burn のみ: 読み返しができないため却下。
- TTL のみ(burn なし): 早く消したい手紙に即応できない。任意フラグとして両立させる。
- TTL の選択UIを提供する: フォームを最小に保つ方針に反するため、v1では既定値を固定とする。
